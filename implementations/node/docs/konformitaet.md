# Conformance — the compatibility contract (Node)

## The idea

Several implementations (PHP reference, Node, later Python) with **identical API and
identical data format**. "Identical" is checkable: a language-neutral **fixture suite**
(pure JSON data). Node counts as conformant when **all fixtures are green** and a
**complete double run yields byte-identical** results — and when the cross-test
against PHP comes out byte-identical.

The fixtures are the normative source. They live in the knowledge base and are mirrored
into `testsuite/` via `make sync` (one-way street). **Never edit here** — see
"SPEC-FINDINGS" below.

## How the runner works

`runner/` implements the contract (`testsuite/README.md`):

1. Build a fresh in-memory tenant from `setup`.
2. Run `steps` in order; `expect.result` by **subset comparison** (only the given
   fields), `expect.error` exactly against the error code.
3. Run `projections` and compare (amounts exact, order normalized).
4. **Double run**: the whole suite twice; traces must be identical after UUID normalization.

Placeholders (`$T1`, `$V1`, `$E1`, …) reference IDs generated by the implementation across
steps. A new subject (e.g. an HTTP client) implements only `runner/src/subject.ts`.

Two subjects, the same runner:

```bash
pnpm fixtures --strict                     # in-memory core
pnpm fixtures --strict --subject=database  # Knex adapter (better-sqlite3)
```

**Cross-test (SF-15)** — the proof of format parity across the language boundary:

```bash
make cross   # PHP↔Node: same data set, journalExport byte-identical, both directions
```

## Determinism rules (the most common cross-impl pitfalls)

From `determinismus.md` in the knowledge base — each has its own fixtures:

- **Commercial half-up rounding** (away from zero at .5): `1.225 → 1.23`. *Not* banker's
  (which would give 1.22). In Node: `big.js` with `Big.roundHalfUp`.
- **VAT per `taxRoundingGranularity`** — `perVoucher`: sum the net per rate, compute tax,
  round **once**; `perLine`: per line item. Comes from the `packPolicy`, not hardwired.
- **`allocate` largest-remainder**, ties → first part in stable order:
  `100.00 / 3 = 33.34 / 33.33 / 33.33`. Σ parts = original amount, always.
- **Sort by Unicode code points** (no locale collation): leading zeros significant,
  `"0420" < "1200" < "8400"`, `"10" < "9"`.
- **Zoneless calendar dates** for the booking date (no UTC shift) — never `new Date()` with a time part.
- **Canonical JSON (RFC 8785)** for hashes/comparisons; float is forbidden (amounts as string decimal).

## SPEC-FINDINGS — the escalation path

When spec, fixture and model contradict each other or something is missing: **do not guess, do not
change the fixture.** Instead:

1. Document in [`../SPEC-FINDINGS.md`](../SPEC-FINDINGS.md) (What / Where / chosen behavior /
   proposal), as `NF-…`.
2. Continue building with the next most plausible behavior.
3. The finding flows back via the knowledge base (decision log) and returns as a refined
   spec + new/changed fixtures.

This is not a stopgap but the intended back channel.
