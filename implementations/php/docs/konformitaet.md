# Conformance — the compatibility contract

## The idea

There are meant to be several implementations (PHP, Node, Python) with
**identical API and identical data format**. "Identical" is not a declaration of
intent but checkable: a language-neutral **fixture suite** (pure JSON data). An
implementation counts as conformant when **all fixtures are green** and a
**complete double run yields byte-identical** results.

The fixtures are the normative source. They live in the knowledge base and are
mirrored into `testsuite/` via `make sync` (one-way street). **Never edit here**
— see "SPEC-FINDINGS" below.

## How the runner works

`runner/` implements the contract (`testsuite/README.md`):

1. Build a fresh in-memory tenant from `setup`.
2. Run `steps` in order; `expect.result` by **subset comparison** (only the
   given fields), `expect.error` exactly against the error code.
3. Run `projections` and compare (amounts exact, ordering normalized).
4. **Double run**: the whole suite twice, traces must be identical after
   UUID normalization.

Placeholders (`$V1`, `$E1`, `$OP1`, …) reference IDs generated by the
implementation across steps.

Two subjects, the same runner:

```bash
php runner/bin/run-fixtures.php --strict                    # in-memory core
php runner/bin/run-fixtures.php --strict --subject=database # database adapter
```

A new subject (e.g. later an HTTP client against Node) only implements
`runner/src/Subject/Subject.php`.

## Determinism rules (the most common cross-impl traps)

From `determinismus.md` of the knowledge base — each has its own fixtures:

- **Commercial rounding half-up** (away from zero at .5): `2.225 → 2.23`.
  *Not* banker's rounding (that would give 2.22).
- **VAT per voucher per tax rate** — form the net sum, compute the tax, round
  **once**. Not per line item.
- **`allocate` largest-remainder**, tie → first part in stable order:
  `100.00 / 3 = 33.34 / 33.33 / 33.33`. Σ parts = original amount, always.
- **Ordering by Unicode code points** (no locale collation): leading zeros are
  significant, `"0420" < "1200" < "8400"`, `"10" < "9"`.
- **Zoneless calendar dates** for the voucher date (no UTC shift).
- **Canonical JSON (RFC 8785)** for hashes/comparisons.

## SPEC-FINDINGS — the escalation path

When spec, fixture, and model contradict each other or something is missing:
**do not guess, do not change the fixture.** Instead:

1. Document it in [`SPEC-FINDINGS.md`](../SPEC-FINDINGS.md) (what / where /
   chosen behavior / proposal).
2. Continue building with the next-most-plausible behavior.
3. The finding flows back through the knowledge base (decision log) and returns
   as a sharpened spec + new/changed fixtures.

This is not a stopgap but the intended back channel — it resolved F-001…F-007
cleanly (history in [`SPEC-FINDINGS.md`](../SPEC-FINDINGS.md)).
